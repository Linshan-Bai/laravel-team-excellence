为什么选择 Pest

PHPUnit 是 PHP 测试的基石，但它的语法冗长，大量样板代码淹没了测试意图。Pest 建立在 PHPUnit 之上，用极简语法重新定义了 PHP 测试体验。

PHPUnit
class OrderTest extends TestCase { public function test_order_total_is_calculated() { $order = Order::factory()->create(); $total = $order->calculateTotal(); $this->assertEquals(100, $total); } }
Pest
it('calculates order total', function () { $order = Order::factory()->create(); expect($order->calculateTotal())->toBe(100); });

Pest 的核心优势

优势说明
零样板代码无需 class、无需 extends TestCase、无需 test_ 前缀
可读性测试名就是一句话，描述业务场景而非技术细节
链式断言expect($value)->toBe()->toBeArray()->toHaveCount() 自然流畅
兼容 PHPUnit100% 兼容 PHPUnit，现有测试无需重写可共存
插件生态Laravel、Livewire、Sanctum 等官方插件开箱即用
并行测试内置 --parallel 支持，测试速度成倍提升
兼容性保证

Pest 是 PHPUnit 的封装层（wrapper），不是替代品。你可以在同一个项目中同时使用 Pest 和 PHPUnit，逐步迁移，无需一次性重写所有测试。

安装与配置

2.1 安装

Bash# Laravel 项目中安装 Pest composer require --dev pestphp/pest pestphp/pest-plugin-laravel # 初始化测试配置（创建 Pest.php） php artisan pest:install

2.2 目录结构

安装后，测试目录遵循 Laravel 约定：

测试目录tests/ ├── Feature/ # 功能测试（HTTP 请求、业务流程） │ ├── OrderTest.php # 也可以用 .test.php 后缀 │ └── AuthTest.php ├── Unit/ # 单元测试（纯逻辑、Service） │ └── CalculatorTest.php ├── Arch/ # 架构测试（依赖关系、分层） │ └── ArchTest.php ├── Pest.php # 全局配置文件 └── TestCase.php # 基础测试用例（Laravel 自动生成）

2.3 全局配置 — Pest.php

tests/Pest.phpuses(Tests\TestCase::class)->in('Feature'); uses(Tests\TestCase::class)->in('Unit'); /*────────────────────────────────────────── │ 全局 beforeEach / afterEach │ 每个测试用例执行前/后自动调用 ──────────────────────────────────────────*/ beforeEach(function () { // 可选：每个测试前清理 }); afterEach(function () { // 可选：每个测试后清理 }); /*────────────────────────────────────────── │ 全局 Macro — 扩展 Expectation API │ 在任何测试文件中使用 expect()->toBeJson() ──────────────────────────────────────────*/ expect()->extend('toBeJson', function () { return $this->toMatchJson(json_encode( json_decode($this->value, true) )); });
关于测试文件后缀

Pest 支持两种后缀：SomethingTest.php（类风格）和 something.test.php（文件风格）。团队统一使用 *Test.php，与 Laravel 约定保持一致。

核心语法速览

3.1 基本测试结构

tests/Feature/BasicTest.phpit('can add two numbers', function () { $result = add(2, 3); expect($result)->toBe(5); }); test('subtraction works correctly', function () { expect(subtract(10, 4))->toBe(6); }); it('throws exception for division by zero', function () { expect(fn () => divide(1, 0)) ->toThrow(InvalidArgumentException::class); });

3.2 数据提供者（Datasets）

Pest 用 with() 替代 PHPUnit 的 @dataProvider，一个测试覆盖多种场景：

tests/Feature/ValidationTest.phpit('validates email format') ->with([ 'user@example.com', 'test+tag@domain.org', 'name.surname@co.uk', ]) ->expect(function (string $email) { expect(isValidEmail($email))->toBeTrue(); }); // 带命名的数据集 it('rejects invalid emails') ->with([ 'missing-at' => 'userexample.com', 'missing-domain' => 'user@', 'has-space' => 'user @example.com', 'empty' => '', ]) ->expect(function (string $email) { expect(isValidEmail($email))->toBeFalse(); });

3.3 测试生命周期钩子

钩子执行时机用途
beforeEach()每个测试前创建通用数据、设置状态
afterEach()每个测试后清理、重置状态
beforeAll()文件内所有测试前（一次）耗时初始化（如创建文件）
afterAll()文件内所有测试后（一次）全局清理
tests/Feature/OrderTest.phpuse App\Models\User; use App\Models\Order; beforeEach(function () { $this->user = User::factory()->create([ 'role' => 'admin', ]); }); it('admin can create order', function () { // $this->user 已自动可用 $response = $this->actingAs($this->user) ->postJson('/api/orders', [ 'product_id' => 1, 'quantity' => 3, ]); $response->assertCreated(); });

3.4 跳过与标记测试

跳过与分组it('pending feature - payment integration') ->skip('Waiting for payment gateway API') ->expect(function () { expect(processPayment(100))->toBeTrue(); }); it('runs only in CI') ->skipUnless(env('CI') === 'true') ->expect(function () { expect(deployed())->toBeTrue(); }); // 按标记运行 it('slow integration test') ->group('slow') ->expect(function () { expect(runFullImport())->toBeTrue(); });

断言系统（Expectation API）

Pest 的 expect() 是链式断言 API，替代 PHPUnit 的 $this->assertEquals() 等方法。以下是团队常用的断言清单：

4.1 常用断言速查

类别断言说明
比较toBe($value)严格相等（===）
toBeTrue() / toBeFalse()布尔断言
toBeNull()必须为 null
toBeGreaterThan($n)大于
类型toBeArray()必须是数组
toBeString()必须是字符串
toBeInt()必须是整数
toBeInstanceOf($class)类型检查
集合/字符串toHaveCount($n)元素数量
toContain($value)包含某个值
toHaveKey($key)数组包含键
toBeEmpty()为空
异常toThrow($class)抛出指定异常
toThrowMessage($msg)异常消息匹配
not()->toThrow()不抛异常
链式组合and($value)切转到另一个值继续断言
not()取反
toBe()（可多次调用）连续断言多个值

4.2 链式断言示例

链式断言expect($user) ->toBeInstanceOf(User::class) ->name->toBe('John') ->email->toContain('@'); expect($response->json('data.orders')) ->toBeArray() ->toHaveCount(3) ->each->toHaveKey('id'); expect(collect($items)) ->not()->toBeEmpty() ->first()->status->toBe('active');

4.3 自定义 Expectation Macro

在 Pest.php 中注册自定义断言，团队共享：

tests/Pest.php// 自定义：断言金额为正数 expect()->extend('toBePositiveAmount', function () { expect($this->value)->toBeNumeric(); expect($this->value)->toBeGreaterThan(0); return $this; }); // 使用 expect($order->total)->toBePositiveAmount();

Laravel 测试集成

5.1 测试基础

安装 pestphp/pest-plugin-laravel 后，Pest 自动获得 Laravel 所有测试能力：actingAs()、withoutMiddleware()、seed() 等。

5.2 使用 Factory

tests/Feature/UserTest.phpuse App\Models\User; use App\Models\Team; beforeEach(function () { $this->user = User::factory()->has( Team::factory()->count(2) )->create([ 'name' => 'Alice', 'email' => 'alice@example.com', ]); }); it('user belongs to teams', function () { expect($this->user->teams)->toHaveCount(2); expect($this->user->name)->toBe('Alice'); });

5.3 Mail / Event / Notification Fake

tests/Feature/RegistrationTest.phpuse App\Mail\WelcomeMail; use App\Events\UserRegistered; use Illuminate\Support\Facades\Mail; use Illuminate\Support\Facades\Event; it('sends welcome email after registration', function () { Mail::fake(); Event::fake(); $this->postJson('/api/register', [ 'name' => 'Alice', 'email' => 'alice@example.com', 'password' => 'Password123!', ])->assertCreated(); // 断言邮件被发送 Mail::assertSent(WelcomeMail::class, function ($mail) { return $mail->hasTo('alice@example.com'); }); // 断言事件被触发 Event::assertDispatched(UserRegistered::class); });

数据库测试

6.1 数据库迁移与事务

Laravel 测试默认在每个测试后自动回滚数据库事务，无需手动清理。

6.2 RefreshDatabase Trait

tests/Pest.phpuses(Illuminate\Foundation\Testing\RefreshDatabase::class)->in('Feature');
RefreshDatabase vs DatabaseMigrations

RefreshDatabase 使用事务回滚（快），DatabaseMigrations 每次重新运行迁移（慢）。团队统一使用 RefreshDatabase。

6.3 数据库断言

tests/Feature/OrderTest.phpit('creates order in database', function () { $this->postJson('/api/orders', [ 'product_id' => 1, 'quantity' => 5, ])->assertCreated(); // 断言数据库中有这条记录 $this->assertDatabaseHas('orders', [ 'product_id' => 1, 'quantity' => 5, 'status' => 'pending', ]); # 断言不存在 $this->assertDatabaseMissing('orders', [ 'status' => 'cancelled', ]); });

6.4 使用 Seed 数据

Seeding// 在单个测试中使用 seed it('shows roles from database', function () { $this->seed(RoleSeeder::class); $response = $this->getJson('/api/roles'); $response->assertOk() ->assertJsonCount(3, 'data'); }); # 全局 seed（Pest.php 中） beforeEach(function () { $this->seed(DatabaseSeeder::class); });

HTTP 测试实战

HTTP 测试是 Laravel 测试中最常用的类型，覆盖 API 和 Web 路由。

7.1 完整的 CRUD 测试模板

tests/Feature/Api/ProductTest.phpuse App\Models\Product; use App\Models\User; beforeEach(function () { $this->admin = User::factory()->create(['role' => 'admin']); $this->user = User::factory()->create(['role' => 'user']); }); // ──── Index ──── it('lists all products', function () { Product::factory()->count(3)->create(); $this->actingAs($this->user) ->getJson('/api/products') ->assertOk() ->assertJsonCount(3, 'data') ->assertJsonStructure([ 'data' => [ '*' => ['id', 'name', 'price'] ] ]); }); // ──── Show ──── it('shows a single product', function () { $product = Product::factory()->create(); $this->actingAs($this->user) ->getJson("/api/products/{$product->id}") ->assertOk() ->assertJsonPath('data.name', $product->name); }); // ──── Store ──── it('admin creates a product', function () { $data = [ 'name' => 'Widget', 'price' => 2999, ]; $this->actingAs($this->admin) ->postJson('/api/products', $data) ->assertCreated() ->assertJsonPath('data.name', 'Widget'); $this->assertDatabaseHas('products', $data); }); // ──── Authorization ──── it('non-admin cannot create products', function () { $this->actingAs($this->user) ->postJson('/api/products', [ 'name' => 'Widget', 'price' => 2999, ]) ->assertForbidden(); }); // ──── Validation ──── it('validates required fields', function () { $this->actingAs($this->admin) ->postJson('/api/products', []) ->assertUnprocessable() ->assertJsonValidationErrors(['name', 'price']); });

7.2 JSON 响应断言速查

断言方法说明
assertOk()HTTP 200
assertCreated()HTTP 201

assertForbidden()HTTP 403
assertNotFound()HTTP 404
assertUnprocessable()HTTP 422
assertJsonPath('data.id', 5)精确路径值

测试文件组织

8.1 目录结构最佳实践

推荐目录tests/ ├── Feature/ │ ├── Auth/ │ │ ├── LoginTest.php │ │ ├── RegisterTest.php │ │ └── PasswordResetTest.php │ ├── Api/ │ │ ├── ProductTest.php │ │ ├── OrderTest.php │ │ └── UserTest.php │ └── HomePageTest.php ├── Unit/ │ ├── Services/ │ │ ├── OrderServiceTest.php │ │ └── PaymentServiceTest.php │ └── ValueObjects/ │ └── MoneyTest.php ├── Arch/ │ └── ArchTest.php ├── Pest.php └── TestCase.php

8.2 测试命名规范

推荐
it('creates order when data is valid') it('rejects invalid email format') it('calculates discount for VIP users') it('sends notification on status change') it('requires authentication for access')
避免
it('test_order') it('it works') it('checks email') it('test123') it('test_vip_discount_function')
AAA 模式

每个测试遵循 Arrange（准备）→ Act（执行）→ Assert（断言） 模式。把三部分用空行隔开，测试结构一目了然。

覆盖率与 CI

9.1 查看覆盖率

Bash# 生成 HTML 覆盖率报告 php artisan test --coverage # 最低覆盖率门槛（低于则失败） php artisan test --min=80 # 查看特定目录覆盖率 php artisan test --coverage --filter=Services # HTML 报告输出 php artisan test --coverage-html=coverage-report

9.2 CI 中的覆盖率报告

.github/workflows/quality.yml（测试部分） - name: Run Pest Tests run: php artisan test --parallel --coverage --min=80 - name: Upload Coverage if: always() uses: actions/upload-artifact@v4 with: name: coverage-report path: coverage-report/

9.3 并行测试

Bash# 自动检测 CPU 核心数，并行执行 php artisan test --parallel # 指定进程数 php artisan test --parallel --processes=4 # 并行 + 覆盖率 php artisan test --parallel --coverage
团队覆盖率目标

目标：> 80%，核心业务模块（支付、权限、订单）要求 > 95%。CI 中设置 --min=80 作为最低门槛，PR 无法低于此标准合并。

测试最佳实践

10.1 测试铁律

原则说明示例
测试行为，不测试实现 关心"做了什么"而非"怎么做的" ✅ 验证返回值 vs ❌ 验证内部变量
一个测试一个断言 测试失败时能立即定位问题 ✅ 一段逻辑一组测试 vs ❌ 一个测试 20 个断言
测试之间完全隔离 不依赖其他测试的状态 ✅ 每个测试独立建数据 vs ❌ 依赖上一个测试的 ID
测试名就是文档 不需要额外注释说明测试意图 ✅ "rejects expired coupon" 自文档化
避免测试框架代码 不要测试 Laravel 自带功能 ❌ 测试 Eloquent save() 方法是否存入数据库
FIRST 原则 Fast / Independent / Repeatable / Self-validating / Timely 测试应该快、独立、可重复、自验证、及时写

10.2 什么不需要测试

  • Laravel 框架自带功能（Eloquent ORM、路由匹配、中间件管道）
  • 第三方包的功能（Stripe SDK、AWS SDK）
  • 简单的 Getter/Setter 和纯数据类
  • Controller 中的依赖注入绑定（框架已测试）

10.3 优先级排序

🔴 高优先级 — 支付、权限验证、价格计算、订单状态流转 🟡 中优先级 — API 接口契约、数据验证、邮件/通知触发 🟢 低优先级 — 管理后台 CRUD、静态页面渲染、日志记录

常见陷阱与避坑指南

1. 不用 fake 直接发真实请求

HTTP 测试中调用外部 API（Stripe、SMS）会发送真实请求，导致扣费或限流。
修复：使用 Http::fake()、Mail::fake()、Event::fake()。

2. 测试依赖执行顺序

测试 A 创建了用户，测试 B 假设用户存在——单独运行测试 B 会失败。
修复：每个测试用 beforeEach() 或 Factory 创建自己的数据。

3. 忽略测试环境配置

测试连接了生产数据库或真实的缓存/队列驱动。
修复：phpunit.xml 中使用 <env name="DB_DATABASE" value="testing"/>，确保 APP_ENV=testing。

4. 过度使用 RefreshDatabase

对大量测试使用 RefreshDatabase（事务回滚）在并行测试时可能冲突。
修复：使用 --parallel 时确保每个测试进程使用独立数据库（Laravel 自动处理）。

5. 测试名描述"怎么做"而非"是什么"

it('uses array_filter to remove nulls') — 实现细节改变后测试名就过时了。
修复：it('removes null values from collection') — 关注结果。

6. 一个文件放太多测试

一个文件 50+ 个测试，难以维护和定位。
修复：按功能拆分子目录（Auth/LoginTest.php、Auth/RegisterTest.php），每个文件 5-20 个测试。

Pest 速查清单

✅ 每次提交前必做

  • composer test — 确保所有测试通过
  • composer test --parallel — 并行运行验证无冲突
  • 新功能必须包含对应测试
  • Bug 修复必须先写失败的测试（红 → 绿 → 重构）

✅ 代码审查时检查

  • 测试名是否描述了业务行为？
  • 是否使用了 Mail::fake() / Http::fake()？
  • 是否有不必要的 Mock（过度 Mock = 测试与实现耦合）？
  • 覆盖率是否达标（核心模块 > 95%）？
