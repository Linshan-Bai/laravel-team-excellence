代码质量工具链分为三层：格式化（让代码看起来一样）、静态分析（在运行前发现错误）、测试（验证行为正确）。本页聚焦前两层。

Laravel Pint — 代码格式化

2.1 安装

Bashcomposer require --dev laravel/pint

2.2 即开即用

Pint 内置 Laravel 预设，零配置即可使用。核心命令：

命令说明
./vendor/bin/pint扫描并自动修复所有文件
./vendor/bin/pint --test只检查，不修改文件（CI 模式）
./vendor/bin/pint app/Models只处理指定目录
./vendor/bin/pint --dirty只处理 Git 变更的文件
./vendor/bin/pint -v显示详细输出
./vendor/bin/pint files/*.php处理指定文件

2.3 自定义规则 — pint.json

项目根目录创建 pint.json，在 Laravel 预设基础上增加团队规则：

pint.json{ "preset": "laravel", "rules": { "strict_comparison": true, "explicit_string_variable": true, "fully_qualified_strict_types": true, "no_superfluous_phpdoc_tags": { "allow_mixed": true }, "ordered_imports": { "sort_algorithm": "alpha", "imports_order": ["const", "class", "function"] }, "single_line_after_imports": true, "trailing_comma_in_multiline": true, "native_function_invocation": { "include": ["@all"] }, "declare_strict_types": true, "blank_line_after_opening_tag": true }, "exclude": ["vendor", "storage", "bootstrap/cache"] }

2.4 修复效果示例

修复前
<?php use App\Models\Order; use Illuminate\Support\Collection; class OrderService { public function getActive($status){ if($status=='active'){ return Order::where('status', $status)->get(); } } }
修复后
<?php declare(strict_types=1); use App\Models\Order; use Illuminate\Support\Collection; class OrderService { public function getActive(string $status): Collection { if ($status === 'active') { return Order::where('status', $status)->get(); } } }

2.5 Pre-commit 集成

在提交代码前自动运行 Pint，防止不规范的代码进入仓库：

.git/hooks/pre-commit#!/bin/sh # 只检查 Git 暂存区的 PHP 文件 ./vendor/bin/pint --dirty if [ $? -ne 0 ]; then echo "\n❌ Pint 发现格式问题，已自动修复。请重新 git add 并提交。" exit 1 fi

PHPStan — 静态分析

3.1 什么是 PHPStan

PHPStan 在不运行代码的情况下，通过分析类型推断来发现 bug。它能捕获以下问题：

  • 类型不匹配（传了 string 给期望 int 的参数）
  • 未定义的属性和方法调用
  • 可能为 null 的值未检查
  • 死代码（永远不会执行的分支）
  • 返回值类型与声明不符

3.2 Level 等级

PHPStan 有 0-9 共 10 个级别。团队目标：Level 9（最严格）。

0-4
基础类型检查
发现基本的类型错误
5-7
严格模式
检查更复杂的类型推断
8-9
最严格
团队目标 Level 9
渐进式策略

如果现有代码无法一步到位 Level 9，建议分阶段提升：
第 1 周：先到 Level 5 → 第 2 周：到 Level 7 → 第 3 周：到 Level 9
使用 --level=5 逐步加严，不要一次跳太猛。

3.3 安装与配置

Bashcomposer require --dev phpstan/phpstan

在项目根目录创建 phpstan.neon：

phpstan.neonparameters: level: 9 paths: - app - config - database - routes bootstrapFiles: - vendor/autoload.php excludePaths: - vendor - storage - bootstrap/cache - database/migrations/*_create_*_table.php checkMissingIterableValueType: false treatPhpDocTypesAsCertain: true reportUnmatchedIgnoredErrors: false ignoreErrors: - message: "#^Call to an undefined method Illuminate\\Support\\HigherOrderTapProxy - message: "#^Access to an undefined property

3.4 常用命令

命令说明
./vendor/bin/phpstan analyse使用 phpstan.neon 配置运行分析
./vendor/bin/phpstan analyse --level=5临时使用 Level 5 运行
./vendor/bin/phpstan analyse app/Services只分析指定目录
./vendor/bin/phpstan analyse --error-format=githubGitHub Actions 格式输出
./vendor/bin/phpstan analyse --memory-limit=512M增大内存限制
./vendor/bin/phpstan clear-result-cache清除缓存（修改配置后）

3.5 典型错误修复

以下是 PHPStan Level 9 最常报告的问题及修复方法：

问题 1：未检查 null

PHPStan 报错
function getUser(int $id): User { return User::findOrFail($id); } $user = getUser(1); echo $user->name; // ⚠️ findOrFail 可能抛异常
修复后
function getUser(int $id): User { return User::findOrFail($id); } echo getUser(1)->name; // ✅ 链式调用，类型明确

问题 2：数组键可能不存在

PHPStan 报错
$data = $request->all(); $name = $data['name']; // ⚠️ Offset 'name' might not exist
修复后
$name = $request->input('name'); // ✅ 使用 FormRequest 方法 if ($request->has('name')) { $name = $request->string('name'); } // ✅ 显式检查后使用

问题 3：严格比较

报错
if ($status == 'active') { ... } // ⚠️ == 可能发生类型转换
修复后
if ($status === 'active') { ... } // ✅ 始终使用 === 严格比较

Larastan — Laravel 专用分析

4.1 为什么需要 Larastan

原生 PHPStan 不理解 Laravel 的魔法——Facade、Eloquent 魔术方法、依赖注入容器等。Larastan 为这些特性添加了类型支持：

Laravel 特性PHPStan 行为Larastan 行为
Auth::user()❌ 报错：静态方法不存在✅ 返回 ?User
User::where('active', true)->get()❌ 找不到 where()✅ 返回 Builder
$user->orders❌ 属性不存在✅ 返回 Collection<Order>
app(SomeService::class)❌ 类型推断失败✅ 返回容器解析类型
View::make('view')❌ 不认识✅ 返回 View

4.2 安装与配置

Bash# Larastan 会自动安装 PHPStan 作为依赖 composer require --dev nunomaduro/larastan

更新 phpstan.neon，引入 Laravel 扩展：

phpstan.neon（完整版）includes: - vendor/nunomaduro/larastan/extension.neon parameters: level: 9 paths: - app - config - database - routes bootstrapFiles: - vendor/autoload.php excludePaths: - vendor - storage - bootstrap/cache checkMissingIterableValueType: false checkGenericClassInNonGenericObjectType: false treatPhpDocTypesAsCertain: true reportUnmatchedIgnoredErrors: false ignoreErrors: - message: "#^Call to an undefined method.*Builder - message: "#^Access to an undefined property

4.3 Model 增强类型

为 Eloquent Model 添加属性类型，让 Larastan 理解你的数据库结构：

app/Models/User.phpclass User extends Authenticatable { protected $fillable = ['name', 'email', 'password']; protected $casts = [ 'email_verified_at' => 'datetime', 'password' => 'hashed', 'is_active' => 'boolean', 'settings' => 'array', ]; /** @return HasMany<Order> */ public function orders(): HasMany { return $this->hasMany(Order::class); } }
Pro Tip

使用 php artisan ide-helper:generate 自动生成 IDE 辅助文件和完整的类型提示。配合 barryvdh/laravel-ide-helper 包使用效果更佳。

PHP CodeSniffer — PSR-12 检查

5.1 定位

CodeSniffer 与 Pint 的区别：Pint = 修复工具，CodeSniffer = 检查工具。CodeSniffer 不修改代码，只报告不符合规范的地方，适合作为 CI 中的最后防线。

5.2 安装

Bash# 安装 CodeSniffer composer require --dev squizlabs/php_codesniffer # 注册编码标准 ./vendor/bin/phpcs --config-set installed_paths vendor/phpcompatibility/php-compatibility

5.3 使用命令

命令说明
./vendor/bin/phpcs --standard=PSR12 app/检查 app 目录是否符合 PSR-12
./vendor/bin/phpcbf --standard=PSR12 app/自动修复 PSR-12 问题
./vendor/bin/phpcs -n app/只显示错误，不显示警告
./vendor/bin/phpcs --report=summary app/摘要报告格式

5.4 创建自定义规则集

phpcs.xml.dist<?xml version="1.0"?> <ruleset name="LaravelTeam"> <rule ref="PSR12"/> <file>app/</file> <file>config/</file> <file>database/</file> <file>routes/</file> <exclude-pattern>vendor/</exclude-pattern> <exclude-pattern>storage/</exclude-pattern> <exclude-pattern>bootstrap/cache/</exclude-pattern> <rule ref="Generic.PHP.DisallowShortOpenTag.EchoFound"/> <rule ref="Generic.Arrays.DisallowLongArraySyntax.Found"/> <property name="lineLimit" value="120"/> <property name="absoluteLineLimit" value="150"/> </ruleset>

IDE 集成与开发体验

6.1 PhpStorm 配置

  1. 安装插件 — 设置 → 插件 → 搜索 "PHPStan" 安装
  2. 配置路径 — 设置 → PHP → Quality Tools → PHPStan → 路径指向 vendor/bin/phpstan
  3. 自动分析 — 勾选 "Run on every save"，保存文件时自动检查
  4. Real-time — 代码编写时实时在编辑器中高亮显示错误

6.2 VS Code 配置

.vscode/settings.json{ "phpstan.enabled": true, "phpstan.config": "./phpstan.neon", "editor.formatOnSave": true, "[php]": { "editor.defaultFormatter": "open-southeners.laravel-pint" }, "laravel-pint.env": { "APP_ENV": "testing" } }

需要安装的 VS Code 扩展：

  • Laravel Pint（open-southeners.laravel-pint）— 保存时自动格式化
  • PHP Intelephense（bmewburn.vscode-intelephense-client）— 智能补全和跳转
  • PHPStan（calebporzio.better-phpstan）— 实时静态分析

6.3 IDE Helper 增强

Bashcomposer require --dev barryvdh/laravel-ide-helper php artisan ide-helper:generate # 生成 IDE 辅助文件 php artisan ide-helper:models # Model 属性补全 php artisan ide-helper:meta # Facade 类型支持

CI 集成方案

7.1 GitHub Actions 配置

.github/workflows/quality.ymlname: Quality Check on: push: branches: [main, develop] pull_request: branches: [main] jobs: quality: runs-on: ubuntu-latest strategy: fail-fast: true steps: - uses: actions/checkout@v4 - name: Setup PHP 8.3 uses: shivammathur/setup-php@v2 with: php-version: '8.3' extensions: mbstring, xml, ctype, iconv, intl - name: Install Dependencies run: composer install --prefer-dist --no-progress - name: Laravel Pint (Format Check) run: ./vendor/bin/pint --test - name: PHPStan Analyse run: ./vendor/bin/phpstan analyse --error-format=github - name: Run Tests run: php artisan test --parallel
CI 检查流程

每次 Push 或 PR 自动运行：Pint 格式检查 → PHPStan Level 9 → Pest 测试套件。任何一步失败都会阻止合并。

Composer Scripts 统一管理

将所有工具命令统一到 composer.json 的 scripts 中，团队成员只需记住 composer xxx：

composer.json（scripts 部分）{ "scripts": { // ========== 格式化 ========== "pint": "./vendor/bin/pint", "pint:test": "./vendor/bin/pint --test", "pint:dirty": "./vendor/bin/pint --dirty", // ========== 静态分析 ========== "stan": "./vendor/bin/phpstan analyse", "stan:5": "./vendor/bin/phpstan analyse --level=5", "stan:7": "./vendor/bin/phpstan analyse --level=7", // ========== 测试 ========== "test": "php artisan test", "test:coverage": "php artisan test --coverage", "test:parallel": "php artisan test --parallel", // ========== 综合检查 ========== "check": [ "@pint:test", "@stan", "@test" ], "check:fast": [ "@pint:dirty", "@stan:5" ] } }
场景命令说明
日常开发composer pint:dirty只修复当前改动的文件，速度快
提交前composer check:fast快速检查格式 + Level 5 分析
PR 前完整检查composer check格式 + Level 9 分析 + 全部测试
只运行测试composer test运行 Pest 测试套件

常见问题排查

9.1 PHPStan 内存溢出

Bash# 增大内存限制 ./vendor/bin/phpstan analyse --memory-limit=1G # 清除缓存 ./vendor/bin/phpstan clear-result-cache

9.2 Laravel Facade 报错

确保 phpstan.neon 包含了 Larastan 的 extension.neon，且配置了正确的 bootstrapFiles。

9.3 Pint 与 PHPStan 冲突

Pint 修改代码后可能引入 PHPStan 新问题。正确顺序：先运行 Pint 修复格式，再运行 PHPStan 检查逻辑。

9.4 暂时忽略特定错误

行内忽略foreach ($items as $item) { echo $item['name']; // @phpstan-ignore-line } // 忽略下一行 // @phpstan-ignore-next-line echo $undefined;
谨慎使用 @phpstan-ignore

每个 @phpstan-ignore 都应该是一个 TODO。定期审查这些忽略标记，真正的解决方式是修复代码而非忽略警告。设置 reportUnmatchedIgnoredErrors: true 可以发现已经不需要的忽略标记。
