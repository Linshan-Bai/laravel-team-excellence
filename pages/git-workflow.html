分支策略

团队采用 简化版 Git Flow，核心原则：保持 main 分支始终可部署，所有工作在分支上进行。

1.1 分支模型

main 生产环境
develop 开发主线
feature/* 功能分支
hotfix/* 紧急修复

1.2 分支说明

分支命名来源合并到说明
main main — — 生产分支，禁止直接推送，仅通过 PR 合入
develop develop main main 开发主线，所有 feature 合入此分支
feature feature/TASK-ID-简述 develop develop 新功能开发，完成后通过 PR 合入 develop
fix fix/TASK-ID-简述 develop develop 非紧急 Bug 修复
hotfix hotfix/TASK-ID-简述 main main + develop 生产环境紧急修复，需同时合入 main 和 develop
release release/v1.2.0 develop main + develop 发布准备（可选），测试通过后合入 main 打 Tag

1.3 分支命名示例

推荐
feature/T-123-user-avatar-upload feature/T-456-order-export-pdf fix/T-789-login-redirect-loop hotfix/T-999-payment-timeout
避免
my-branch update-stuff fix-bug dev test feature1
保护分支

在 GitHub 仓库 Settings → Branches 中启用 Branch Protection Rules：main 分支要求：PR Review（至少 1 人）、CI 通过、禁止 force push。

提交信息规范（Conventional Commits）

团队统一使用 Conventional Commits 规范。好处：自动生成 Changelog、语义化版本控制、快速定位变更类型。

2.1 提交格式

<type>(<scope>): <description> # 可选：详细说明（换行后写） # 可选：关联的 Issue / Task Refs: T-123

2.2 Type 类型

Type说明影响版本示例
feat 新功能 MINOR feat(auth): add OAuth2 login
fix Bug 修复 PATCH fix(order): correct tax calculation
refactor 重构（不改变行为） — refactor(svc): extract payment logic
docs 文档更新 — docs(readme): update install guide
test 测试相关 — test(order): add CRUD test suite
chore 构建/工具变更 — chore(deps): upgrade Laravel to 11.x
perf 性能优化 PATCH perf(query): add eager loading to orders
style 格式化（不影响逻辑） — style: fix PSR-12 formatting

2.3 提交信息示例

推荐
feat(order): add PDF export for invoices - Generate PDF using DomPDF - Support Chinese character rendering - Include company logo and signature Refs: T-234
避免
update code fix stuff aaaaaaa WIP merge branch done

2.4 Scope 参考

Scope 表示变更影响范围，常用值：

auth user order payment product api svc db ui deps ci docs

开发工作流

3.1 日常开发流程

完整工作流# 1. 从 develop 创建功能分支 git checkout develop git pull origin develop git checkout -b feature/T-123-user-avatar-upload # 2. 开发 → 提交 git add -p # 交互式暂存，审查每个改动 git commit -m "feat(user): add avatar upload endpoint" # 3. 推送到远程 git push -u origin feature/T-123-user-avatar-upload # 4. 创建 Pull Request（develop ← feature） # 在 GitHub 上操作，填写 PR 模板 # 5. Code Review → CI 通过 → 合并 # 6. 清理分支 git checkout develop git pull origin develop git branch -d feature/T-123-user-avatar-upload git push origin --delete feature/T-123-user-avatar-upload

3.2 紧急修复流程

Hotfix 流程# 从 main 创建 hotfix 分支 git checkout main git pull origin main git checkout -b hotfix/T-999-payment-timeout # 修复 → 提交 → 推送 git add . git commit -m "fix(payment): handle gateway timeout" git push -u origin hotfix/T-999-payment-timeout # 创建 PR 到 main，Review 通过后合并 # 合并到 main 后，同步到 develop git checkout develop git pull origin develop git merge main git push origin develop
注意

Hotfix 必须同时合入 main 和 develop，否则修复会丢失在下一个版本中。

Git Hooks 自动化

Git Hooks 让特定命令（commit、push、PR）自动触发检查，在代码进入仓库前拦截问题。

4.1 Hooks 一览

pre-commit

提交前运行 Pint 格式化 + PHPStan 检查 + 测试，不通过则阻止提交

最常用

commit-msg

验证提交信息是否符合 Conventional Commits 格式

推荐

pre-push

推送前运行完整检查（Pint + Stan + Test），防止问题推到远程

可选

prepare-commit-msg

自动在提交信息中添加分支号（如 T-123），减少手动输入

可选

Pre-commit 配置

推荐使用 Laravel Pint 自带 Hook 或 pre-commit 框架 管理 Hooks。

5.1 方案一：Laravel Pint 内置（推荐，最简单）

Bash# Pint 自带 Git Hook 安装命令 ./vendor/bin/pint install

这会在 .git/hooks/pre-commit 中写入 Pint 的 Hook。提交时自动格式化。

5.2 方案二：自定义综合 Hook（推荐，覆盖更全）

.git/hooks/pre-commit#!/bin/sh echo "\n🔍 Running Pre-commit Checks...\n" # ── Step 1: Laravel Pint（格式化） ── echo "📝 Step 1/3: Laravel Pint..." ./vendor/bin/pint --dirty if [ $? -ne 0 ]; then echo "\n❌ Pint 修复了格式问题。请重新 git add 并提交。" exit 1 fi # ── Step 2: PHPStan Level 5（快速分析） ── echo "🔬 Step 2/3: PHPStan (Level 5)..." ./vendor/bin/phpstan analyse --level=5 --no-progress 2>/dev/null if [ $? -ne 0 ]; then echo "\n❌ PHPStan 发现类型错误。请修复后再提交。" exit 1 fi # ── Step 3: Pest（快速测试，只跑变更相关） ── echo "🧪 Step 3/3: Pest Tests..." php artisan test --parallel 2>/dev/null if [ $? -ne 0 ]; then echo "\n❌ 测试失败。请修复后再提交。" exit 1 fi echo "\n✅ All checks passed! Committing...\n"

5.3 方案三：使用 pre-commit 框架（大型项目推荐）

Bash# 安装 pre-commit 框架（Python，跨语言） pip install pre-commit brew install pre-commit # macOS # 创建配置文件
.pre-commit-config.yamlrepos: - repo: local hooks: - id: laravel-pint name: Laravel Pint entry: ./vendor/bin/pint language: script types: [php] pass_filenames: false - id: phpstan-fast name: PHPStan (Level 5) entry: ./vendor/bin/phpstan analyse --level=5 --no-progress language: script pass_filenames: false always_run: true - id: pest-tests name: Pest Tests entry: php artisan test --parallel language: script pass_filenames: false always_run: true
Bash# 安装 hooks 到 .git/ pre-commit install # 手动运行所有 hooks pre-commit run --all-files

5.4 commit-msg Hook（提交信息验证）

.git/hooks/commit-msg#!/bin/sh # 读取提交信息的第一行 MSG=$(head -n1 "$1") # 验证 Conventional Commits 格式 if ! echo "$MSG" | grep -qE "^(feat|fix|refactor|docs|test|chore|perf|style|ci|build|revert)(\(.+\))?!?: .+"; then echo "❌ 提交信息格式错误！" echo "" echo "正确格式：" echo " type(scope): description" echo "" echo "类型：feat|fix|refactor|docs|test|chore|perf|style|ci" echo "示例：feat(auth): add OAuth2 login" exit 1 fi

Changelog 自动生成

有了 Conventional Commits，可以自动生成变更日志。

6.1 使用 conventional-changelog

Bash# 全局安装 npm install -g conventional-changelog-cli # 生成 Changelog（基于上次 Tag 以来的提交） conventional-changelog -p angular -i CHANGELOG.md -s # 生成完整 Changelog（所有提交） conventional-changelog -p angular -i CHANGELOG.md -s -r 0

6.2 生成效果预览

# v1.3.0 (2026-04-03) ### Features * **auth**: add OAuth2 Google login (#42) * **order**: add PDF export for invoices (#38) ### Bug Fixes * **payment**: handle gateway timeout (#45) * **order**: correct tax calculation (#40)

Code Review 流程

7.1 PR 模板

在仓库根目录创建 .github/pull_request_template.md：

.github/pull_request_template.md## 变更说明 ## 变更类型 - [ ] feat（新功能） - [ ] fix（Bug 修复） - [ ] refactor（重构） - [ ] docs（文档） - [ ] test（测试） - [ ] chore（构建/工具） ## 关联任务 Refs: T-XXX ## 测试情况 - [ ] 新增了对应测试 - [ ] 所有现有测试通过 - [ ] 本地手动验证通过 ## 自查清单 - [ ] 代码遵循 PSR-12 规范 - [ ] 没有硬编码的配置值 - [ ] 没有调试代码（dd/dump/console.log） - [ ] 数据库变更包含迁移文件 - [ ] 新增了必要的 PHPDoc 注释

7.2 Review 评审要点

维度检查项
正确性逻辑是否正确？边界条件是否处理？
安全性是否有 SQL 注入/XSS/权限漏洞风险？
性能是否有 N+1 查询？是否有不必要的数据库操作？
可读性命名是否清晰？是否需要注释？
测试是否有对应测试？覆盖率是否足够？
规范是否符合 PSR-12？类型提示是否完整？

7.3 Review 反馈原则

  • 对代码不对人 — "这段逻辑可能导致 X 问题" vs "你写的有问题"
  • 区分阻塞和建议 — 阻塞问题（Must Fix）vs 优化建议（Nice to Have）
  • 说明原因 — "建议改成 X，因为..." 而非只说 "改一下"
  • 及时响应 — PR 在 24 小时内完成 Review

合并策略

8.1 三种合并方式对比

策略命令历史推荐场景
Merge Commit git merge --no-ff 保留完整分支历史 feature → develop
Squash & Merge GitHub Squash 压缩为一条提交 小功能 PR（< 5 个 commit）
Rebase & Merge git rebase + merge 线性历史 个人分支同步 develop

8.2 团队约定

合并规则
  • feature → develop：使用 Merge Commit（--no-ff），保留功能开发上下文
  • develop → main：使用 Squash 或 Merge Commit
  • hotfix → main：使用 Merge Commit
  • 同步 develop：个人分支用 git rebase origin/develop 保持线性
  • 禁止：对已推送到远程的公共分支执行 git push --force

常见问题排查

9.1 合并冲突

Bash# 查看冲突文件 git status # 手动编辑冲突文件，解决 <<<<<<< HEAD / >>>>>>> markers # 标记为已解决 git add . git commit -m "merge: resolve conflicts with develop" # 如果冲突太多，重新开始 git merge --abort

9.2 误提交到 main

Bash# 撤销最后一次提交（保留改动） git reset --soft HEAD~1 # 撤销并创建功能分支 git checkout -b feature/T-xxx-fix git push -u origin feature/T-xxx-fix
危险操作警告

如果误提交的内容已经推送到远程 main，不要使用 git push --force。应该通过 Revert 或联系管理员处理。Force push 公共分支是严重事故。

9.3 Pre-commit Hook 跳过（紧急情况）

Bash# ⚠️ 仅在紧急 hotfix 时使用 git commit --no-verify -m "hotfix: emergency fix" # 稍后补运行检查 composer check

速查清单

Git 工作流速查

📋 每日必做

  • git pull origin develop — 开始工作前同步最新代码
  • 从 develop 创建功能分支，不要直接在 develop 上开发
  • 提交前确保 Pre-commit 通过（composer check:fast）
  • 使用 Conventional Commits 格式：feat|fix|refactor(scope): desc

📋 PR 前

  • composer check — 完整检查（Pint + Stan Level 9 + Test）
  • 填写 PR 模板所有字段
  • 自查清单全部打勾

📋 合并后

  • 删除远程和本地功能分支
  • 同步 develop：git checkout develop && git pull

🚫 坚决不做

  • ❌ 直接 push 到 main（必须通过 PR）
  • ❌ 在公共分支上 --force push
  • ❌ 提交 .env 文件、密钥、密码等敏感信息
  • ❌ 提交包含 dd() / dump() 的调试代码
  • ❌ 频繁使用 --no-verify 跳过 Hook
