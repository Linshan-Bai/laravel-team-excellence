为什么需要代码规范

代码规范不是教条——它是一份团队契约。统一的编码风格让代码像一个人写的，降低认知负担，减少 Code Review 中的格式争论。

规范带来的好处
  • 降低认知负担 — 阅读代码时不需要适应不同人的风格
  • 减少 Review 噪音 — 格式问题交给工具，Review 专注逻辑
  • 新人快速上手 — 统一风格 = 统一期望
  • 自动化检查 — 配合 Laravel Pint 实现零人工干预

Laravel 生态遵循 PSR-12（PHP-FIG 扩展编码风格），在此基础上我们增加团队约定。

命名规范

2.1 概览表

元素 风格 示例

2.2 布尔变量和方法

布尔变量/方法使用 is、has、can、should 等前缀，让意图不言自明：

推荐
$isActive = true; $hasPermission = false; if ($user->canEdit()) { ... } if ($order->hasItems()) { ... } if ($this->shouldRetry()) { ... }
避免
$active = true; $permission = false; if ($user->edit()) { ... } if ($order->items()) { ... } if ($this->retry()) { ... }

2.3 避免缩写和单字母

推荐
$userAccount $paymentMethod $orderRepository function getActiveUsers()
避免
$usrAcc $payMeth $orderRepo function getUsers()
例外

常见缩写可接受：使用 $id 而非 $identifier（太长），$url 而非 $uniformResourceLocator（过分）。循环计数器 $i、$j 也可以。

严格类型提示

PHP 8.3 + Laravel 11 完全支持类型系统。团队约定：所有方法参数和返回值都必须声明类型。

3.1 方法签名

推荐 — 完整类型提示
public function getActiveUsers( string $role, int $limit = 20, ): Collection { return $this->user ->where('role', $role) ->where('is_active', true) ->limit($limit) ->get(); }
避免 — 无类型声明
public function getActiveUsers( $role, $limit = 20, ) { return $this->user ->where('role', $role) ->where('is_active', true) ->limit($limit) ->get(); }

3.2 属性类型声明

推荐
class CreateOrderRequest extends FormRequest { public readonly string $customerName; public readonly int $quantity; public readonly ?string $couponCode; public readonly CarbonImmutable $orderedAt; }
避免
class CreateOrderRequest extends FormRequest { public $customerName; public $quantity; public $couponCode; public $orderedAt; }

3.3 Nullable 与联合类型

PHP 8.3public function findUser(int $id): ?User { return User::find($id); // 可能返回 null } public function resolveValue(string|int $key): string|int|null { return $this->cache[$key] ?? null; } public function filter( callable(int): bool $callback, array<int> $items, ): array { ... }
实用技巧
  • 数组元素类型用泛型注释 /** @param array<User> $users */
  • 集合返回值写 @return Collection<int, User>
  • declare(strict_types=1) 放在文件最顶部

3.4 Enum 类型（PHP 8.1+）

Laravel 11 全面支持枚举，用 Enum 替代魔术字符串和整数常量：

推荐 — 使用 Enum
enum OrderStatus: string { case Pending = 'pending'; case Paid = 'paid'; case Shipped = 'shipped'; case Cancelled = 'cancelled'; } public function updateStatus( OrderStatus $status ): void { ... }
避免 — 魔术字符串
public function updateStatus( string $status ): void { if ($status === 'pending') { ... } if ($status === 'paid') { ... } // 拼写错误不会报错！ }

注释规范

4.1 核心原则：代码 > 注释

好的代码自己会说话。注释应该解释 Why（为什么），而不是 What（做了什么）。

推荐 — 代码即文档
// 计算到期日：合同到期前 30 天通知 public function getNotificationDate(): Carbon { return $this->expires_at->subDays(30); } // 使用二分查找而非线性搜索 // 因为数据集已排序且超过 10000 条 $result = $this->binarySearch($items, $target);
避免 — 废话注释
// 获取用户名 $name = $user->name; // 循环遍历订单 foreach ($orders as $order) { // 处理每个订单 $this->process($order); } // 返回 true return true;

4.2 DocBlock 注释

以下场景 必须 写 DocBlock：

  • 类的用途说明
  • 复杂方法的参数/返回值（类型提示不够直观时）
  • 公共 API 接口
  • 包含复杂业务逻辑的地方
DocBlock 示例/** * 计算订单折扣金额 * * 应用多层折扣规则：会员折扣 → 满减优惠 → 优惠券 → 限时折扣 * 折扣按优先级从高到低依次叠加，最终折扣不超过商品总价 * * @param Order $order 需要计算折扣的订单 * @return DiscountResult 包含折扣明细和最终价格的结果对象 * * @throws InsufficientStockException 当库存不足时 * @throws ExpiredCouponException 当优惠券已过期时 */ public function calculateDiscount(Order $order): DiscountResult

4.3 TODO / FIXME

// TODO: 添加缓存层，当前每次都查数据库 (Linshan 2026-04-01) // FIXME: 并发场景下可能出现超卖问题，需加分布式锁
注意

每个 TODO/FIXME 必须附上负责人和日期。超过 30 天的 TODO 应在 Sprint 回顾中讨论——要么解决，要么删掉。

方法设计原则

5.1 单一职责

每个方法只做一件事。方法长度控制在 20 行以内，超过就拆分。

推荐 — 拆分为多个小方法
public function processOrder(Order $order): OrderResult { $this->validateStock($order); $this->applyDiscount($order); $this->reserveInventory($order); $payment = $this->processPayment($order); return OrderResult::from($order, $payment); }
避免 — 一个方法做所有事
public function processOrder(Order $order) { foreach ($order->items as $item) { if ($item->stock < $item->qty) { throw new ...; } } if ($order->coupon) { // 50 行折扣计算... } // 还有 100 行... }

5.2 参数数量

方法参数不超过 3 个。超过时使用 DTO（数据传输对象）或数组：

推荐 — 使用 DTO
public function createUser( CreateUserDTO $data, ): User { ... } // DTO 中包含 name, email, // password, role 等所有字段
避免 — 参数过多
public function createUser( string $name, string $email, string $password, string $role, bool $isActive, ?string $avatar, ): User { ... }

5.3 尽早返回（Guard Clauses）

推荐 — 扁平化逻辑
public function getPrice(?Product $product): int { if ($product === null) { return 0; } if ($product->isFree()) { return 0; } if ($product->isOnSale()) { return $product->salePrice(); } return $product->price; }
避免 — 深层嵌套
public function getPrice(?Product $product): int { if ($product !== null) { if (!$product->isFree()) { if ($product->isOnSale()) { return $product->salePrice(); } else { return $product->price; } } } return 0; }

5.4 避免布尔参数

布尔参数让调用处的代码不可读：$service->send(true, false) 谁知道这是什么意思？

推荐 — 语义清晰的方法
// 调用处一目了然 $notifier->sendWithRetry($order); $notifier->sendWithoutRetry($order); $notifier->sendSilently($order);
避免
// 什么意思？ $notifier->send($order, true, false);

类与文件结构

6.1 文件规则

  • 每个文件只包含一个类
  • 文件名与类名完全一致：UserController.php
  • 声明 declare(strict_types=1) 在最顶部
  • 命名空间与目录结构一致：App\Services\Payment
  • 使用 use 引入类，不写完整命名空间
文件模板<?php declare(strict_types=1); namespace App\Services\Payment; use App\Models\Order; use Illuminate\Support\Facades\Log; use RuntimeException; /** * 处理支付相关业务逻辑 */ class PaymentService { public function __construct( private readonly PaymentGateway $gateway, private readonly Logger $logger, ) {} public function process(Order $order): PaymentResult { // ... } }

6.2 类成员排列顺序

保持一致的排列顺序能帮助快速定位代码：

  1. use 语句 — 按字母排序，分三组（PHP 内置 / 第三方 / 应用内）
  2. 常量 — public const
  3. 属性 — 先 public 再 protected 最后 private
  4. 构造函数 — __construct()
  5. 公共方法
  6. 受保护方法
  7. 私有方法
use 语句分组
use RuntimeException; // PHP 内置 use Illuminate\Support\Collection; // 框架 use Carbon\CarbonImmutable; // 第三方 // 空行分隔 use App\Models\User; // 应用内 use App\Services\NotificationService;

EditorConfig 配置

EditorConfig 确保所有编辑器和 IDE 使用相同的缩进和格式。在项目根目录创建 .editorconfig：

.editorconfigroot = true [*] charset = utf-8 end_of_line = lf indent_size = 4 indent_style = space insert_final_newline = true trim_trailing_whitespace = true [*.md] trim_trailing_whitespace = false [*.{yml,yaml,json}] indent_size = 2 [*.blade.php] indent_size = 4 [composer.json] indent_size = 4
编辑器支持

VS Code、PhpStorm、Sublime Text 等主流编辑器都原生支持 EditorConfig。团队成员 clone 项目后自动生效，无需手动配置。

PHP-CS-Fixer 自动化

Laravel Pint 底层使用 PHP-CS-Fixer。以下是 Pint 的配置方法：

8.1 安装 Laravel Pint

Bashcomposer require --dev laravel/pint

8.2 配置 pint.json

在项目根目录创建 pint.json，定义团队规则集：

pint.json{ "preset": "laravel", "rules": { "strict_comparison": true, "explicit_string_variable": true, "fully_qualified_strict_types": true, "no_superfluous_phpdoc_tags": { "allow_mixed": true }, "ordered_imports": { "sort_algorithm": "alpha", "imports_order": [ "const", "class", "function" ] }, "single_line_after_imports": true, "trailing_comma_in_multiline": true, "native_function_invocation": { "include": ["@all"] }, "declare_strict_types": true, "blank_line_after_opening_tag": true } }

8.3 Composer Script 集成

在 composer.json 中添加快捷命令：

composer.json{ "scripts": { "pint": "./vendor/bin/pint", "pint:test": "./vendor/bin/pint --test", "check": [ "@pint:test", "./vendor/bin/phpstan analyse" ] } }

8.4 常用命令

命令说明
./vendor/bin/pint自动修复所有文件
./vendor/bin/pint --test只检查不修改（CI 用）
./vendor/bin/pint app/Services只检查指定目录
./vendor/bin/pint --dirty只检查 Git 修改的文件
composer pint通过 Composer 运行

速查清单

日常开发中的快速对照表，建议打印或设为桌面便签：

必须做到
  • ✅ declare(strict_types=1) 在所有 PHP 文件顶部
  • ✅ 所有方法参数和返回值都有类型声明
  • ✅ 类名 PascalCase，方法/变量 camelCase
  • ✅ 布尔变量用 is/has/can/should 前缀
  • ✅ 方法不超过 20 行，参数不超过 3 个
  • ✅ 用 Guard Clause 减少嵌套
  • ✅ 使用 Enum 替代魔术字符串
  • ✅ 注释解释 Why 而不是 What
  • ✅ .editorconfig 配置到位
  • ✅ 提交前运行 composer pint
坚决不做
  • ❌ 不使用类型声明的方法
  • ❌ 魔术字符串 / 魔术数字（if ($status === 3)）
  • ❌ 超过 3 层的 if-else 嵌套
  • ❌ 超过 50 行的方法
  • ❌ 不写注释的复杂业务逻辑
  • ❌ 中英文混用的命名（$yongHuMing）
  • ❌ 单字母变量名（循环计数器除外）
